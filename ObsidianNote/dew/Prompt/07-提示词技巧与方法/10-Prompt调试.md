# Prompt 调试

> Prompt 不是一次就能写好的——就像代码需要调试，Prompt 也需要系统性地测试和迭代优化。

---

## Prompt 模板

```
# Prompt 调试流程

## Step 1：准备测试用例
- 准备 5-10 个有代表性的输入
- 覆盖正常情况和边界情况

## Step 2：写第一版 Prompt
- 能跑就行，不追求完美

## Step 3：全量测试
- 用所有测试用例测试，记录每个结果

## Step 4：分析失败案例
- 找出共性问题（格式不对？逻辑错？编造事实？）

## Step 5：修改 Prompt
- 一次只改一处，控制变量

## Step 6：重新测试
- 确认修改有效，没有引入新问题

## Step 7：重复 Step 4-6 直到满意
```

---

## 常见失败模式速查表

| 失败模式 | 可能原因 | 修复方法 |
|----------|----------|----------|
| AI 不遵循格式 | 示例不够、要求不明确 | 加更多示例、用"只返回"、用 XML/JSON Schema |
| AI 编造事实 | 任务超出知识范围 | 加"如果不知道就说不知道"、引入检索 |
| 输出太长/太短 | 长度要求不明确 | 明确字数要求、给示例长度 |
| 逻辑错误 | 任务太难，直接跳步 | 加入思维链、拆分成多步 |
| 风格不对 | 角色设定不清晰 | 加更详细的 System Prompt、给风格示例 |

---

## 技巧

- **控制变量**：一次只改一处——改了两个地方，你不知道是哪个起了作用
- **版本管理**：给每个版本起名（v1, v2, v2.1），记录改了什么、为什么改、通过率多少
- **保留旧版**：效果好的 Prompt 先备份再改，可以随时回滚
- **黄金法则**：如果一个 Prompt 工作得很好，不要因为"看起来可以更好"就去改它
- **对比测试**：新旧版本用同一批测试用例跑，对比通过率再决定是否替换
